# Vox Companion Bridge Spec

## Purpose

Define how the Linea website communicates with an installed local companion
(`Vox Companion`) that can run native ASR and alignment workloads using local
GPU / ANE acceleration.

The companion is an enhancement layer. The website must continue to function
without it.

## Product framing

- The website remains the primary UI.
- Vox Companion is a local execution runtime.
- The companion is optional.
- If the companion is available, Linea uses it automatically.
- If the companion is unavailable, Linea degrades gracefully to cloud or
  heuristic behavior.

## Initial scope

Phase 1 of the companion focuses on local transcription and alignment only.

Primary use case:

- TTS audio is generated by OpenAI or ElevenLabs.
- Vox Companion transcribes the generated clip locally.
- The transcription produces precise word timestamps.
- Linea uses those timestamps for playback highlighting, seeking, and document
  markers.

Non-goals for Phase 1:

- Local TTS generation
- Full offline product mode
- Local document ingestion or PDF parsing
- General-purpose plugin execution

## Architecture

Communication uses a loopback bridge on `127.0.0.1`.

The stack is:

- Browser: Linea website running at `https://app.uselinea.com`
- Companion: local app or daemon with native ML runtime
- Transport: localhost HTTP for probing and job setup, WebSocket for live
  session updates

Deep links are used only to launch or focus the companion. Deep links are not
the primary transport for active sessions.

## Design principles

1. The site must never assume the companion exists.
2. The companion must be discoverable via capability probing.
3. The website must use one stable browser-facing API regardless of backend.
4. Local acceleration should improve the experience, not gate the experience.
5. The localhost bridge must reject arbitrary origins by default.

## Runtime model

The companion installs a local service that listens on a fixed loopback port.

Recommended default:

- `http://127.0.0.1:43115`

The local service may be hosted by:

- a minimal native menu bar app
- a background daemon launched by a host app
- an existing Vox runtime with a thin installer and launcher

## Connection lifecycle

### 1. Probe

On page load, the website probes the local service:

- `GET /health`
- `GET /capabilities`

If the probe succeeds, the UI may show `Vox Companion connected`.

If the probe fails, the UI should remain usable and offer:

- `Install companion`
- `Open companion`

### 2. Authenticate

If the companion is available, the browser establishes a session token or local
handshake before opening an active WebSocket session.

### 3. Session

The website opens a WebSocket connection for low-latency events:

- `ws://127.0.0.1:43115/session`

This channel carries:

- session state
- job start acknowledgement
- progress events
- completion events
- errors

### 4. Execute

The website submits an alignment job referencing generated audio and source
metadata.

### 5. Fallback

If the companion disconnects or declines the job, Linea falls back to:

- cloud alignment, if configured
- heuristic timing otherwise

## Security model

The companion must not behave like an open localhost API.

Minimum requirements:

- Only accept requests from approved origins
- Default allowlist includes `https://app.uselinea.com`
- Require a short-lived session token or challenge handshake
- Reject requests with missing or invalid session credentials
- Bind only to `127.0.0.1`, never `0.0.0.0`

Recommended first-pass security:

1. `GET /health` returns public status only
2. `GET /capabilities` returns non-sensitive capability metadata
3. `POST /session/token` or equivalent local handshake issues a short-lived
   token
4. WebSocket upgrade to `/session` requires:
   - approved `Origin`
   - valid short-lived token

Optional future hardening:

- first-run user approval per origin
- per-account binding
- signed job payloads from Linea server

## Capability model

The website should adapt based on what the companion actually supports.

Example capability payload:

```json
{
  "running": true,
  "version": "0.1.0",
  "features": {
    "alignment": true,
    "local_asr": true,
    "streaming_progress": true
  },
  "backends": {
    "parakeet": true,
    "mlx": true,
    "ane": true
  }
}
```

Capabilities should describe facts, not product promises.

## HTTP endpoints

### `GET /health`

Purpose:

- lightweight liveness check

Example response:

```json
{
  "ok": true,
  "service": "vox-companion",
  "version": "0.1.0"
}
```

### `GET /capabilities`

Purpose:

- discover whether the companion supports the requested workflow

Example response:

```json
{
  "running": true,
  "version": "0.1.0",
  "features": {
    "alignment": true,
    "local_asr": true,
    "streaming_progress": true
  },
  "backends": {
    "parakeet": true,
    "mlx": true,
    "ane": true
  }
}
```

### `POST /jobs`

Purpose:

- create a local processing job

Example request:

```json
{
  "type": "alignment",
  "sessionId": "sess_123",
  "source": {
    "audioUrl": "https://app.uselinea.com/api/vox/audio/abc123",
    "format": "mp3"
  },
  "metadata": {
    "documentId": "doc_123",
    "pageNumber": 2,
    "paragraphId": "page-2-paragraph-4"
  }
}
```

Example response:

```json
{
  "jobId": "job_123",
  "accepted": true
}
```

### `GET /jobs/:jobId`

Purpose:

- fetch latest terminal state or recover after reconnect

## WebSocket protocol

WebSocket endpoint:

- `ws://127.0.0.1:43115/session`

### Session start

Browser to companion:

```json
{
  "type": "session.start",
  "token": "short_lived_token",
  "sessionId": "sess_123"
}
```

Companion to browser:

```json
{
  "type": "session.ready",
  "sessionId": "sess_123",
  "capabilities": {
    "alignment": true
  }
}
```

### Job progress

Companion to browser:

```json
{
  "type": "job.progress",
  "jobId": "job_123",
  "stage": "transcribing",
  "progress": 0.42,
  "message": "Running Parakeet locally"
}
```

Stages should be implementation-agnostic. Suggested values:

- `accepted`
- `fetching_audio`
- `preparing`
- `transcribing`
- `aligning`
- `finalizing`

### Job completion

Companion to browser:

```json
{
  "type": "job.complete",
  "jobId": "job_123",
  "result": {
    "alignment": {
      "words": [
        { "word": "Hello", "start": 0.12, "end": 0.41 }
      ],
      "durationMs": 12800
    }
  }
}
```

### Job failure

Companion to browser:

```json
{
  "type": "job.error",
  "jobId": "job_123",
  "code": "backend_unavailable",
  "message": "Parakeet runtime is not available"
}
```

## Job semantics

For Phase 1, the key job type is `alignment`.

Input:

- audio reference
- optional source metadata
- session context

Output:

- word-level timestamps
- duration

The browser should treat alignment as optional enhancement data.

If alignment fails:

- playback must continue
- the UI may show a non-blocking state change
- highlighting falls back to estimated timing

## Audio source strategy

Preferred first implementation:

- the website provides a signed audio URL
- the companion fetches the audio directly

Advantages:

- avoids moving large audio blobs through the browser twice
- keeps browser memory pressure lower
- works well for cached generated clips

Fallback option:

- browser uploads audio bytes directly to the local companion

This may still be useful for:

- local-only generated previews
- future offline flows

## UX behavior

### When companion is available

- show connected state quietly
- use precise alignment automatically
- avoid making the user choose between transports

### When companion is unavailable

- keep voice playback working
- do not show hard errors just because local alignment is missing
- offer installation or launch affordances in setup UI

### When companion is installed but not running

- show `Open companion`
- retry local probe after launch

## Versioning

The companion must report a semantic version string in:

- `GET /health`
- `GET /capabilities`

The website should use capability checks, not version checks, whenever
possible.

## Failure handling

The browser must handle these cases explicitly:

- localhost probe failed
- session handshake failed
- companion disconnected mid-job
- job accepted but processing failed
- companion returned partial capability set

Expected behavior:

- log diagnostics
- fall back safely
- keep user-visible errors specific and non-blocking

## Open questions

1. Should Linea server issue signed short-lived job tokens for companion jobs,
   or should the local handshake remain fully local?
2. Should the companion fetch audio from signed URLs, or should the website
   upload audio blobs directly in Phase 1?
3. Should the companion run continuously in the background, or only after a
   user explicitly opens it?
4. Do we want first-run permission approval per domain before `app.uselinea.com`
   can open a session?

## Recommendation

Ship the first version with:

- localhost HTTP probe
- localhost WebSocket session
- alignment-only job support
- signed audio URL fetch by the companion
- strict origin allowlist
- graceful website fallback when the companion is unavailable

That gives Linea the benefits of native ASR acceleration without making the
website dependent on local installation.
